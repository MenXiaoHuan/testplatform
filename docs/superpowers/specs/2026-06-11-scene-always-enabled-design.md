# 场景恒定启用设计

## 背景

当前平台的场景列表同时承载了三个概念：

- 场景是否启用
- 场景是否可执行
- 场景的管理操作

随着仓库启停能力和执行按钮图标化调整落地，业务语义已经收敛为：

- 场景本身不存在“启用/停用”概念
- 场景默认始终可被管理
- 是否允许点击执行，只取决于所属代码仓库是否启用

继续保留场景 `enabled` 字段和列表开关，会造成页面心智和后端模型长期不一致。

## 目标

将“场景默认始终开启”收敛为平台统一规则，删除场景启停概念，并同步清理前后端和数据库中的历史字段。

## 非目标

- 不调整场景定时能力，`scheduleEnabled` 和 `cronExpression` 继续保留
- 不调整仓库启停语义，仓库仍然可以启用或停用
- 不调整任务执行和报告解析链路

## 目标体验

### 场景列表

- 删除场景开关
- 第一列保留为 `执行控制`
- `执行控制` 列只展示“执行图标”
- `操作` 列仅展示 `查看任务`、`编辑`、`删除`
- 执行图标在所属仓库停用时置灰不可点击

### 场景创建和编辑

- 场景不再包含启停状态
- 前端表单和保存请求不再读写 `enabled`
- 列表和详情接口不再依赖 `enabled`

## 前端设计

### 类型收敛

更新以下类型，移除场景 `enabled` 字段：

- `SceneRecord`
- `SceneDetail`
- `SceneForm`

默认表单创建逻辑不再生成 `enabled`。

### 列表页面

文件：

- `playwright-platform-web/src/views/scene/SceneListView.vue`

改动：

- 去掉列表开关
- 删除 `toggleEnabled()` 及其调用链
- 删除 `togglingId`
- `执行控制` 列只保留执行图标
- 执行图标禁用逻辑：
  - 所属仓库不存在：禁用
  - 所属仓库停用：禁用
  - 其他情况：可执行
- `操作` 列保持纯文本一行展示

### store 和保存载荷

文件：

- `playwright-platform-web/src/stores/scene.ts`
- `playwright-platform-web/src/types/scene.ts`

改动：

- 不再向后端提交 `enabled`
- 不再要求回填 `enabled`
- 现有保存逻辑继续保留其他字段不变

### 前端测试

需要更新或新增测试覆盖：

- 场景列表不再出现开关
- 列名为 `执行控制`
- 执行按钮为图标入口
- 仓库停用时执行图标禁用
- 操作列仅保留 `查看任务 / 编辑 / 删除`

## 后端设计

### 实体与接口收敛

文件范围：

- `SceneEntity`
- `SceneServiceImpl`
- `SceneController`
- 相关测试

改动：

- 删除 `SceneEntity.enabled`
- 场景创建和更新不再读写 `enabled`
- 场景列表和详情返回不再包含 `enabled`

### 调度逻辑

当前调度扫描语义需要从“启用且开启定时”调整为“开启定时”。

保留逻辑：

- 只扫描 `scheduleEnabled = true` 的场景

删除逻辑：

- 不再基于场景 `enabled` 做过滤

### 执行逻辑

任务执行入口无需新增后端字段判断。

前端已经限制“仓库停用时不可点执行”，后端继续保留现有仓库有效性校验，避免绕过前端时误执行。

## 数据库设计

新增 migration：

- 删除 `scene.enabled` 列

同时同步更新测试初始化 SQL 和相关 fixture，确保不会继续依赖该列。

## 兼容性

这是一次模型收敛，不保留场景启停兼容层。

原因：

- 业务规则已经明确“场景一定是开启的”
- 前端继续隐藏、后端继续保留会制造长期歧义
- 数据库字段不再有保留价值

## 风险与控制

### 风险

- 场景测试和控制器测试会因字段删除而失败
- 数据库 migration 之后，本地旧数据初始化脚本可能需要同步调整
- 如果遗漏某个接口序列化字段，前后端模型会短期漂移

### 控制

- 先更新测试，再改实现
- 聚焦回归场景模块相关前后端测试
- 对场景列表页和创建/编辑表单做手工验证

## 验证清单

### 前端

- 场景列表不再显示开关
- `执行控制` 列仅展示执行图标
- 所属仓库停用时执行图标禁用
- 场景新增和编辑正常保存

### 后端

- 创建场景成功
- 更新场景成功
- 查询列表和详情成功
- 定时扫描逻辑仍能正常工作

### 数据库

- migration 执行成功
- 场景表中不再有 `enabled` 列

## 实施顺序

1. 更新前端和后端测试，锁定“场景恒定启用”新语义
2. 删除前端 `enabled` 读写和列表开关
3. 删除后端实体、服务、接口中的 `enabled`
4. 新增数据库 migration 并更新测试数据
5. 回归场景相关前后端测试与页面行为
